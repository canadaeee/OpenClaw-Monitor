# 部署与实施约束文档 - OpenClaw Monitor

## 1. 部署目标

OpenClaw Monitor 作为独立的只读观测系统，v1 目标是优先稳定运行在 **OpenClaw 所在本机环境** 中，并以 **本机访问、本机采集、node 侧旁路观测** 为首要路径。

系统需要同时满足以下要求：

- 服务端适合长期运行。
- 被监控对象可以来自 Linux、macOS、Windows。
- 观测页面默认面向本机浏览器访问，不依赖跨设备访问 Gateway。
- 在需要时可通过固定端口提供受控访问，但这不是 v1 首要目标。

## 2. 总体部署策略

### 2.1 推荐部署形态

v1 推荐使用以下部署方式：

- **服务端部署系统**：优先 Ubuntu，其次为正在运行 OpenClaw node 的本机系统
- **部署位置**：OpenClaw node 本机优先，其次才是 Gateway 同机或同网段主机
- **访问方式**：本机浏览器访问 Monitor Web 页面
- **跨端访问方式**：保留为后续增强，不作为 v1 首发必需能力

### 2.2 跨平台原则

Monitor 采用“**核心系统统一、采集层按平台适配**”的设计：

- 后端服务统一开发
- 前端页面统一开发
- 事件模型统一
- API 契约统一
- Linux / macOS / Windows 仅在采集与系统指标获取层存在差异

不分别为不同操作系统开发三套 Monitor。

## 3. 系统组成

### 3.1 服务端

服务端运行在 OpenClaw node 本机或 Gateway 本机上，负责：

- 接收事件与心跳
- 存储原始事件与标准化数据
- 执行状态推导
- 执行告警判断
- 提供只读 API
- 托管前端静态页面

### 3.2 被观测端

被观测端可以是：

- Ubuntu / Linux 节点
- macOS 节点
- Windows 节点

它们可以通过以下方式接入：

- 主动上报事件
- 主动上报心跳
- 日志文件采集
- 资源指标采集

在 v1 中，推荐优先采用 **node 本机采集**：

- Monitor 与 OpenClaw node 运行在同一台机器
- 直接采集本机日志、事件文件、任务产物元数据或本地进程输出
- 通过本机 collector 转换为标准事件，再写入 Monitor

如果 node 已经与 Gateway 完成配对，则 Monitor 不要求重复参与 Gateway 的设备配对流程。

### 3.3 访问端

访问端不限定操作系统，可包括：

- Windows 工作站
- macOS 笔记本
- Linux 桌面
- 其他能访问浏览器的终端

访问端只需能够访问 Ubuntu 上暴露的 Monitor 端口，无需本地安装完整服务端。

## 4. 观测窗口跨端访问设计

## 4.1 设计原则

观测窗口在 v1 中应优先作为 **本机 Web 界面** 运行，而不是把“远程跨设备访问”作为首要目标。

这意味着：

- Monitor 前端应由本机服务端托管。
- 默认使用本机浏览器访问 `127.0.0.1:12888`。
- 不要求为了远程访问而复刻 OpenClaw Control UI 的设备身份机制。

## 4.2 适配你的使用场景

对于“Ubuntu 是服务器，Windows 通过 Tailscale + SSH 长期连接”的使用方式，推荐两种访问方案：

### 方案 A：本机访问

适用场景：

- OpenClaw 与 Monitor 运行在同一台机器
- 当前目标是稳定完成本机观测

推荐方式：

- Monitor 默认监听 `127.0.0.1:12888`
- 本机浏览器直接访问 `http://127.0.0.1:12888`
- node 侧 collector 从本机日志、事件或进程输出采集数据

这是 v1 的推荐主路径。

### 方案 B：直接通过 Tailscale 访问 Monitor 端口

适用场景：

- Ubuntu 与 Windows 都加入同一 Tailscale 网络
- 允许 Ubuntu 暴露 Monitor 监听端口给 Tailscale 内网访问

推荐方式：

- Monitor 监听固定端口，例如 `127.0.0.1:12888` 或 `0.0.0.0:12888`
- 通过 Tailscale ACL 或 Ubuntu 防火墙控制访问来源
- Windows 浏览器直接访问 Ubuntu 的 Tailscale 地址加端口

示例：

- `http://100.x.y.z:12888`
- 或使用 MagicDNS 域名访问

### 方案 B：通过 SSH 本地端口转发访问

适用场景：

- 不希望直接对网络暴露 Monitor 服务端口
- 你已长期通过 SSH 登录 Ubuntu

推荐方式：

- Monitor 服务仅监听 Ubuntu 本机回环地址，例如 `127.0.0.1:12888`
- Windows 使用 SSH 端口转发把本地端口映射到 Ubuntu 的 12888
- Windows 浏览器访问本地映射端口

示例概念：

- Windows 本地访问 `http://localhost:12888`
- 由 SSH 把该端口转发到 Ubuntu 的 Monitor 服务

这是在本机模式之外的补充方式，不作为 v1 必须能力。

## 4.3 端口策略

建议在文档和实现里固定一个默认端口，避免每次切换环境都重新找地址。

v1 建议：

- **默认 HTTP 端口**：`12888`
- **默认监听地址**：`127.0.0.1`
- **需要跨设备直接访问时**：改为受控网卡地址或配合反向代理暴露

建议约定：

- 开发环境默认监听 `127.0.0.1:12888`
- 生产 / 常驻环境若需跨端访问，优先通过 Tailscale 或 Nginx/Caddy 反向代理提供入口

## 5. 技术栈建议

### 5.1 后端

v1 固定为：

- **Python**
- **FastAPI**

原因：

- 不因 OpenClaw 本体使用 Swift 而跟随切换语言
- 适合快速构建只读 API 与事件处理服务
- 在 Ubuntu 上部署简单
- 与 SQLite、JSONL、后台任务处理模型匹配度高
- 适合日志解析、文本处理与 node 侧旁路采集

### 5.2 前端

v1 固定为：

- **React**
- 构建后由服务端托管静态资源

不强制使用 Next.js，因为 v1 不需要 SSR 和复杂同构能力。

补充约定：

- v1 前端继续使用当前 React 工程，不因 OpenClaw 控制台技术实现而强制改用其他语言或框架
- 如需增强类型安全，可在后续版本平滑迁移到 TypeScript，但不作为当前切换前提

### 5.3 存储

v1 使用：

- **SQLite**：存储标准化实体、索引与聚合结果
- **JSONL**：存储原始事件流与可审计日志

## 6. 网络与端口约束

### 6.1 服务监听建议

推荐分两种模式：

- **本机安全模式**
  - 监听 `127.0.0.1:12888`
  - 本机浏览器访问
- **内网访问模式**
  - 监听 `0.0.0.0:12888` 或指定内网网卡
  - 通过防火墙、Tailscale ACL、反向代理限制访问

### 6.2 不建议的做法

- 不建议默认直接暴露到公网
- 不建议默认无认证监听所有网卡
- 不建议让前端页面依赖 SSH 终端界面本身渲染

## 7. 安全与访问控制

### 7.1 基础要求

- Monitor 必须支持认证访问。
- 至少应支持单用户登录保护或反向代理层认证。
- 若通过 Tailscale 使用，可结合 Tailscale 网络身份控制访问来源。

### 7.2 数据保护

- 原始事件中若包含敏感信息，必须支持脱敏或最小化存储。
- 不应默认展示完整敏感输入内容。
- 应记录访问日志和关键查询日志，便于审计。

## 8. 实施建议

### 8.1 v1 建议落地方式

建议按以下顺序实施：

1. 在 OpenClaw node 本机启动 FastAPI 服务和前端静态页面。
2. 服务默认监听 `127.0.0.1:12888`。
3. 先通过 node 本机 collector 接入日志、事件文件或进程输出。
4. 确认本机监控页面稳定后，再评估是否通过 Tailscale 或 SSH 转发开放访问。
5. 若后续有多人使用，再增加反向代理、TLS 和认证层。

### 8.2 接入策略

被观测系统应优先提供以下最小接入能力：

- Agent 心跳
- Node 心跳
- 任务生命周期事件
- Token 使用事件
- 错误事件

只要这套最小事件集存在，Monitor 就可以形成基本可用的黑盒可视化能力。

对于已经与 Gateway 配对的 node，v1 推荐优先从 node 本机采集这些事件，而不是要求 Monitor 自身建立新的 Gateway 设备身份。

## 9. 验收标准

### 9.1 部署验收

- Ubuntu 服务端可长期稳定运行。
- 服务端重启后可恢复数据读取与页面访问。
- Windows 终端可通过 SSH 转发或 Tailscale 正常访问观测页面。

### 9.2 访问验收

- 浏览器可以通过固定地址和端口访问 Monitor。
- 页面跨端访问体验一致，不依赖终端系统差异。
- 当端口不可达或服务异常时，能明确区分网络问题与数据问题。

### 9.3 安全验收

- 未授权用户不能直接访问观测页面和 API。
- 未经允许不直接暴露到公网。
- 暴露到内网时具备最小访问控制能力。
