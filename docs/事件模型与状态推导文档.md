# 事件模型与状态推导文档 - OpenClaw Monitor

## 1. 设计目标

OpenClaw Monitor 面向黑盒观测场景，所有页面展示、告警判断和统计分析都应尽量建立在**标准化事件**之上。系统不假设 OpenClaw 能直接提供完整状态，而是通过事件、心跳和时间窗口推导出当前可见状态。

## 2. 事件设计原则

- 每个事件都应可独立存档和重放。
- 同一任务相关的事件必须通过 `trace_id` 和 `task_id` 关联。
- 事件允许不完整，但必须明确缺失字段，不得使用含糊值替代。
- 原始事件和标准化事件可以并存，展示层优先消费标准化事件。

## 3. 公共字段

所有标准化事件至少包含以下字段：

- `event_id`: 事件唯一标识
- `event_type`: 事件类型
- `occurred_at`: 事件发生时间
- `ingested_at`: 事件被 Monitor 接收时间
- `trace_id`: 链路标识；若无链路上下文可为空
- `task_id`: 任务标识；非任务类事件可为空
- `agent_id`: 关联 Agent；若不适用可为空
- `node_id`: 关联 Node；若不适用可为空
- `session_id`: 会话标识；若无可为空
- `source`: 事件来源，例如 `gateway`, `agent`, `node`, `collector`
- `severity`: `info | warning | error | critical`
- `payload`: 事件特有字段对象

## 4. 事件字典

### 4.1 心跳类事件

#### `agent_heartbeat`

表示某个 Agent 仍然在线并上报了最新活动。

`payload` 建议字段：

- `agent_name`
- `version`
- `host`
- `current_task_id`
- `queue_depth`
- `reported_status`

#### `node_heartbeat`

表示某个 Node 仍然在线并上报了基础资源情况。

`payload` 建议字段：

- `node_name`
- `host`
- `cpu_percent`
- `memory_percent`
- `disk_percent`
- `load_average`

### 4.2 任务生命周期事件

#### `task_created`

任务已被创建，但尚未进入执行队列。

`payload` 建议字段：

- `task_type`
- `task_name`
- `creator`
- `priority`

#### `task_queued`

任务已进入队列等待执行。

`payload` 建议字段：

- `queue_name`
- `position`
- `reason`

#### `task_started`

任务开始由某个 Agent 或 Node 执行。

`payload` 建议字段：

- `executor_agent_id`
- `executor_node_id`
- `attempt`
- `started_by`

#### `task_waiting`

任务处于等待资源、等待上游结果或等待外部依赖的状态。

`payload` 建议字段：

- `wait_reason`
- `blocked_by`

#### `task_progress`

任务在执行中产生阶段性进展。

`payload` 建议字段：

- `progress_percent`
- `stage`
- `message`

#### `task_completed`

任务执行完成。

`payload` 建议字段：

- `duration_ms`
- `result_status`
- `artifact_count`

#### `task_failed`

任务执行失败。

`payload` 建议字段：

- `error_code`
- `error_message`
- `retryable`
- `failed_stage`

### 4.3 模型与成本事件

#### `model_invoked`

发生一次模型调用。

`payload` 建议字段：

- `model_name`
- `supplier_name`
- `usage_category`
- `request_id`
- `latency_ms`

#### `token_usage`

模型调用结束后产出的 Token 使用数据。

`payload` 建议字段：

- `model_name`
- `supplier_name`
- `usage_category`
- `token_input`
- `token_output`
- `total_token`
- `cost_estimate`
- `currency`

### 4.4 产物与错误事件

#### `artifact_emitted`

任务执行过程中产生了一个可追踪产物。

`payload` 建议字段：

- `artifact_type`
- `artifact_path`
- `artifact_size`
- `content_type`

#### `error_logged`

记录某个执行过程中的错误信息，未必意味着任务最终失败。

`payload` 建议字段：

- `error_code`
- `error_message`
- `stack_summary`
- `related_component`

### 4.5 告警事件

#### `alert_triggered`

某条告警被触发。

`payload` 建议字段：

- `alert_id`
- `alert_type`
- `severity`
- `dedup_key`
- `related_task_id`
- `related_agent_id`
- `related_node_id`
- `trigger_reason`

#### `alert_resolved`

某条告警恢复。

`payload` 建议字段：

- `alert_id`
- `resolved_reason`
- `resolved_by_rule`

## 5. 状态模型

## 5.1 Agent 状态

Agent 状态分为两层：

### 连接状态

- `online`
- `offline`
- `unknown`

### 运行状态

- `idle`
- `queued`
- `running`
- `waiting`
- `error`
- `unknown`

### 推导规则

- 在离线阈值内收到 `agent_heartbeat`，连接状态为 `online`。
- 连续超过离线阈值未收到 `agent_heartbeat`，连接状态为 `offline`。
- 最近存在未结束的 `task_started` 事件，运行状态为 `running`。
- 最近存在 `task_waiting` 且无后续完成/失败事件，运行状态为 `waiting`。
- 最近存在 `task_failed` 且无后续成功执行信号，可标记为 `error`。
- 若在线但无活动任务，运行状态为 `idle`。
- 数据不足时标记为 `unknown`。

## 5.2 Node 状态

Node 也分为连接状态和运行观察状态。

### 连接状态

- `online`
- `offline`
- `unknown`

### 推导规则

- 在离线阈值内收到 `node_heartbeat`，为 `online`。
- 超过阈值未收到心跳，为 `offline`。
- 缺乏足够数据时为 `unknown`。

## 5.3 Task 状态

Task 使用以下统一枚举：

- `created`
- `queued`
- `running`
- `waiting`
- `completed`
- `failed`
- `timeout`
- `unknown`

### 推导规则

- 收到 `task_created` 后进入 `created`。
- 收到 `task_queued` 后进入 `queued`。
- 收到 `task_started` 后进入 `running`。
- 收到 `task_waiting` 后进入 `waiting`。
- 收到 `task_completed` 后进入 `completed`。
- 收到 `task_failed` 后进入 `failed`。
- 在 `running` 或 `waiting` 状态下超过超时阈值未更新，进入 `timeout`。
- 缺少起始事件但存在孤立日志时标记为 `unknown`。

### 状态优先级

当同一时间窗口内存在冲突事件时，按以下优先级判定最终状态：

`failed / completed / timeout` > `running / waiting` > `queued` > `created` > `unknown`

## 5.4 Alert 状态

- `open`
- `resolved`
- `suppressed`

### 推导规则

- 收到 `alert_triggered` 后进入 `open`。
- 收到对应 `alert_resolved` 后进入 `resolved`。
- 若系统后续支持抑制规则，可进入 `suppressed`。

## 6. 时间窗口建议

建议在 v1 固定以下窗口，避免前后端口径漂移：

- Agent 心跳周期：10 秒
- Agent 离线判定：30 秒无心跳
- Node 心跳周期：10 秒
- Node 离线判定：30 秒无心跳
- 任务超时判定：10 分钟无状态更新
- 短时间错误突增观察窗口：5 分钟

## 7. 告警判定规则

### 7.1 基础告警

- Agent 离线：超过离线阈值未收到 `agent_heartbeat`
- Node 离线：超过离线阈值未收到 `node_heartbeat`
- 任务超时：任务在执行态超过超时窗口未更新
- 任务失败：收到 `task_failed`
- 单任务 Token 异常：`total_token` 超过阈值
- 错误突增：观察窗口内错误事件数量超过阈值

### 7.2 告警字段要求

每条告警至少要具备：

- `alert_id`
- `alert_type`
- `severity`
- `status`
- `triggered_at`
- `resolved_at`
- `dedup_key`
- `related_task_id`
- `related_agent_id`
- `related_node_id`

## 8. 原始事件与标准化事件

系统建议同时保留两种数据：

- **原始事件**：保留源系统输出，便于审计与回溯
- **标准化事件**：由 Monitor 清洗后生成，供查询和页面展示使用

页面默认展示标准化事件，必要时允许查看原始事件内容。
