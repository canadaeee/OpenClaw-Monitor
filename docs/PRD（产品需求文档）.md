# 产品需求文档 (PRD) - OpenClaw Monitor

## 1. 产品定位

**OpenClaw Monitor** 是一个独立于 OpenClaw 主系统之外的只读观测平台，目标是在**不改变 OpenClaw 执行逻辑、不介入任务控制**的前提下，将原本不可见的运行过程转化为可观察、可追踪、可告警的监控视图。

该系统的核心价值不是“管理 OpenClaw”，而是“让 OpenClaw 的黑盒执行过程变得可视化”。它通过采集任务事件、Agent/Node 心跳、模型调用信息、Token 消耗、错误记录与产物元数据，构建统一的观测能力，帮助团队完成以下工作：

- 判断系统当前是否健康。
- 判断某个任务执行到了哪里。
- 判断异常从哪个环节开始出现。
- 判断 Token、成本和资源消耗集中在哪里。

## 2. 产品原则

### 2.1 设计原则

- **只读原则**：Monitor 不提供暂停、取消、重试、重分配等控制能力。
- **旁路原则**：Monitor 作为旁路观测系统存在，不改变 OpenClaw 原有调度链路。
- **事件优先原则**：所有监控状态优先基于事件和心跳推导，而不是依赖源系统内存态。
- **诚实展示原则**：当观测数据不完整时，界面应展示“未知”或“数据缺失”，不能伪造状态。
- **追踪优先原则**：优先保证任务链路可追踪，再补充高级分析能力。

### 2.2 不做的事情

本项目明确不包含以下内容：

- 任务控制与干预。
- Agent 配置管理。
- 调度策略编排。
- 自动修复或自愈动作。
- 对 OpenClaw 主系统执行逻辑的侵入式改造。

## 3. 目标用户

- **管理员 / 运维人员**：关注系统是否健康、是否有离线节点、是否有堆积或超时任务。
- **开发人员**：关注任务执行链路、失败原因、模型调用、Token 消耗和错误上下文。
- **产品经理 / 团队负责人**：关注系统整体活跃度、运行效率、成本变化和异常趋势。

## 4. 核心问题

OpenClaw 当前作为执行系统，其运行过程具有明显黑盒特征：

- 无法快速判断一个任务卡在何处。
- 无法快速定位是 Agent、Node、模型调用还是输入数据导致失败。
- 无法清楚统计 Token 和成本的来源。
- 无法用统一视角观察系统健康度与异常演变过程。

OpenClaw Monitor 的目标是解决以上问题。

## 5. 核心能力

### 5.1 健康可视化

- 展示 Agent、Node 的在线状态与最近活动时间。
- 展示当前运行任务、等待任务、失败任务和超时任务数量。
- 展示今日 Token 消耗、错误数量和活跃告警数量。

### 5.2 任务链路可视化

- 展示任务从创建、排队、执行、等待、完成或失败的完整生命周期。
- 展示任务经过的 Agent、Node、模型调用与产物输出。
- 使用 `trace_id` 串联单任务全链路。

### 5.3 异常定位

- 对任务超时、任务失败、Agent/Node 离线、Token 异常等情况触发告警。
- 提供与告警相关联的任务、Agent、事件和时间线视图。

### 5.4 Token / Cost 可视化

- 按任务、Agent、模型、供应商聚合 Token 使用量。
- 提供按日、周、月的 Token 趋势统计。
- 在条件允许时提供成本估算。

### 5.5 事件查询

- 按任务、时间范围、事件类型、Agent、Node、告警等级筛选事件流。
- 支持查看原始事件与标准化后的聚合视图。

## 6. 三大观测支柱

### 6.1 Metrics

用于快速判断系统整体健康与趋势：

- 在线 Agent 数
- 在线 Node 数
- 运行中任务数
- 等待中任务数
- 失败任务数
- 超时任务数
- 今日 Token 消耗
- 今日错误数
- 活跃告警数

### 6.2 Events / Logs

用于还原单点事实：

- 任务生命周期事件
- Agent / Node 心跳事件
- 模型调用事件
- Token 使用事件
- 错误事件
- 告警触发与恢复事件
- 产物输出事件

### 6.3 Trace

用于还原单个任务的完整执行路径：

- 使用 `trace_id` 关联同一任务的全部事件。
- 展示任务经过的 Agent / Node / Model / Artifact。
- 支持从告警跳转到对应 Trace。

## 7. MVP 范围

### 7.1 v1 必做

- 总览页：展示系统健康、任务分布、Token 总览、活跃告警和最近异常。
- Agent 列表与详情页：展示连接状态、运行状态、最近任务、最近错误、今日 Token。
- Node 列表与详情页：展示连接状态和基础资源信息。
- 任务列表与详情页：展示任务生命周期、执行时间线、关联 Agent/Node、Token 和错误信息。
- 事件流页：支持筛选和检索关键事件。
- 告警页：展示告警触发、恢复、关联对象和当前状态。
- Token / Cost 页：按模型、供应商、Agent、任务查看 Token 聚合和趋势。
- 基于事件和心跳的状态推导能力。

### 7.2 v1 不做

- 任务暂停、取消、重试、重新分配。
- 自定义复杂告警编排。
- 多租户权限体系。
- 自动化运维动作。
- 深度容量预测和预算管理。

### 7.3 v1.1 可扩展

- 更细粒度的告警抑制与确认流程。
- Trace 对比分析。
- 更完整的成本归因。
- 更丰富的 Node 性能指标。

## 8. 页面与用户路径

### 8.1 系统总览

用于回答“现在是否有问题”：

- 系统健康摘要
- 在线 Agent / Node 数
- 当前任务状态分布
- 今日 Token 和错误趋势
- 当前活跃告警
- 最近异常事件

### 8.2 任务追踪

用于回答“这个任务发生了什么”：

- 任务列表
- 任务详情
- 生命周期时间线
- Trace 视图
- 关联错误、产物、模型调用记录

### 8.3 Agent / Node 观察

用于回答“是谁出问题了”：

- Agent 当前运行状态
- 最近活动时间
- 最近失败任务
- Node 在线性和资源占用

### 8.4 Token / Cost 分析

用于回答“资源花到哪里去了”：

- 按模型统计
- 按供应商统计
- 按 Agent 统计
- 按任务统计
- 趋势图

### 8.5 事件与告警

用于回答“证据在哪里”：

- 原始事件流
- 标准化事件视图
- 告警触发 / 恢复记录
- 多条件筛选

## 9. 告警范围

v1 仅包含面向故障定位的基础告警：

- Agent 离线
- Node 离线
- 任务超时
- 任务失败
- 单任务 Token 异常
- 短时间错误突增

以下内容不作为 v1 告警：

- 每日 Token 消耗排名
- 预算优化建议
- 高级异常检测模型

## 10. 成功标准

### 10.1 产品成功标准

- 团队可以在 Monitor 中快速判断系统是否异常。
- 团队可以通过单任务时间线定位主要故障位置。
- 团队可以按 Agent、Node、任务、模型查看 Token 消耗。
- 团队可以通过事件流和告警关联视图完成基本排障。

### 10.2 技术成功标准

- 支持至少 50 个 Agent 和 10 个 Node 的日常观测。
- 状态推导结果与原始事件一致，不出现明显口径冲突。
- 事件查询和列表页面具备基本分页与筛选能力。
- 在源系统数据不完整时，界面能正确展示未知状态而不是误报。

## 11. 落地约束

- Monitor 部署为独立系统，运行于 Ubuntu Gateway。
- 通过心跳、日志、事件上报或文件采集获取数据。
- 优先以最小接入代价完成 v1，不要求 OpenClaw 提供完整控制 API。
- 若 OpenClaw 暂时无法提供全部事件，则以“可观测最小集”先落地，并明确标注缺失项。
