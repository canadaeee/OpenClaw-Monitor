# 分发与部署方案文档 - OpenClaw Monitor

## 1. 目标

本文件用于明确 OpenClaw Monitor 的对外分发方式和安装策略，支持后续面向开发者或用户进行一键部署。

## 2. 推荐方案

推荐采用：

- **代码发布到 GitHub**
- **提供官方一键安装脚本**
- **提供 Docker 部署方式作为补充**

不建议只做其中一种。

## 3. 为什么推荐 GitHub + 官方脚本

### 3.1 GitHub 的价值

- 方便版本管理
- 方便 issue / release / changelog 管理
- 方便用户查看源码和提交问题
- 方便后续接入 CI/CD

### 3.2 官方脚本的价值

- 降低首次部署门槛
- 统一安装路径和默认端口
- 统一依赖安装流程
- 让 Ubuntu 用户可以一条命令完成部署

对于 OpenClaw Monitor 这类内部工具或半内部工具，最佳方式通常不是“只发源码”，而是“源码仓库 + 官方部署脚本”。

## 4. 建议的分发层级

### v1

- GitHub 仓库
- Ubuntu 一键安装脚本
- README 中提供本机访问方式说明
- README 中补充 node 侧 collector 的接入说明

### v1.1

- Docker Compose 部署
- systemd 服务文件
- 配置模板

### v1.2

- 可选发布二进制包或打包发行版

## 5. 一键部署脚本建议

建议做成类似官方安装体验：

### 入口方式

例如：

```bash
curl -fsSL <script-url> | bash
```

Windows PowerShell 可使用：

```powershell
irm <script-url> | iex
Install-OpenClawMonitor -RepoUrl https://github.com/<owner>/<repo>.git
```

或：

```bash
bash install.sh
```

### 脚本职责

- 检查操作系统是否为 Ubuntu / Linux
- 检查 Python / Node / npm 环境
- 拉取或更新代码
- 安装后端依赖
- 安装前端依赖并构建
- 生成默认配置
- 默认监听 `127.0.0.1:12888`
- 生成 node-side collector 示例配置
- 输出访问说明
- 可选写入 systemd 服务
- 支持 bootstrap 脚本先克隆仓库，再调用平台安装脚本

## 6. 推荐仓库结构

```text
OpenClaw-Monitor/
  backend/
  frontend/
  docs/
  scripts/
    install.sh
    start.sh
    stop.sh
    update.sh
  README.md
```

## 7. 推荐结论

最终建议如下：

- **主分发方式**：GitHub
- **主安装方式**：官方一键安装脚本
- **补充方式**：Docker Compose

换句话说，不建议在“脚本”与“GitHub”之间二选一，而是：

**把 GitHub 作为发布载体，把官方脚本作为安装体验。**

这会最接近你提到的 “OpenClaw 官方脚本式一键部署” 的使用感，同时保留标准开源仓库的可维护性。

补充说明：

- v1 默认支持 **本机访问 + 本机 node 侧采集**
- 不把“跨设备访问 Gateway”作为首发硬要求
- 对已与 Gateway 配对的 node，优先通过本机 collector 采集本地事件，而不是让 Monitor 自身重新参与设备配对
